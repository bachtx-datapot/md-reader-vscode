# Event Pattern — Domain Event Conventions

> _meta: source=`shared/events/outbox-producer.ts`, `shared/events/types.ts`, `docs/event-conventions.md` | extracted=2026-02-27

## DomainEvent<T> Envelope

```typescript
// WHY typed envelope: all events conform to this shape for cross-module contracts
export interface DomainEvent<T = Record<string, unknown>> {
  eventId: string       // UUID — auto-generated by outbox table
  eventType: string     // Dot-notation: {module}.{entity}.{action}
  module: string        // Source module name
  tenantId: string      // Tenant isolation key
  correlationId?: string // Optional distributed tracing ID
  timestamp: string     // ISO 8601 — set by outbox createdAt
  payload: T            // Module-specific event data
}
```

## EmitEventInput (producer provides)

```typescript
export interface EmitEventInput<T = Record<string, unknown>> {
  eventType: string      // e.g., 'crm.lead.transitioned'
  payload: T             // Domain-specific data
  module: string         // e.g., 'crm'
  tenantId?: string      // Defaults to 'default'
  correlationId?: string // Optional tracing ID
}
```

## Emitting Events — THE ONLY WAY

```typescript
import { emitOutboxEvent } from '@/shared/events/outbox-producer'

// WHY inside transaction: event committed atomically with business data
await db.transaction(async (tx) => {
  await tx.insert(someTable).values(data)
  await emitOutboxEvent(tx, {
    eventType: 'crm.lead.transitioned',
    payload: { leadId: data.id, transition: 'created' },
    module: 'crm',
    tenantId: data.tenantId,
  })
})

// WHY emitOutboxEvent only: publishEvent() is DEPRECATED
// WHY outbox pattern: prevents lost events if app crashes after DB write
```

## Batch Events

```typescript
import { emitOutboxEvents } from '@/shared/events/outbox-producer'

// WHY batch: single business operation producing multiple events
await emitOutboxEvents(tx, [
  { eventType: 'lms.enrollment.created', payload: {...}, module: 'lms', tenantId },
  { eventType: 'fin.invoice.created', payload: {...}, module: 'fin', tenantId },
])
```

## Direct Emit (no transaction)

```typescript
import { emitOutboxEventDirect } from '@/shared/events/outbox-producer'

// WHY direct: for emitters NOT inside an existing db.transaction()
// Still atomic (single INSERT), but no multi-table atomicity
await emitOutboxEventDirect(db, { eventType: '...', payload: {...}, module: '...', tenantId })
```

## Event Naming Convention

```
{module_prefix}.{entity}.{action}

Examples:
  crm.lead.transitioned      — lead lifecycle (discriminator in payload)
  crm.quote.created           — individual event per action
  fin.invoice.created         — WHY fin. not bil.: migrated 2026-02-21
  lms.enrollment.activated    — enrollment lifecycle
  inf.user.created            — WHY inf. not infra.: prefix matches table prefix
```

## Transition Event Pattern (discriminated payload)

```typescript
// WHY single event with discriminator: KISS — fewer event types to manage
await emitOutboxEvent(tx, {
  eventType: 'crm.lead.transitioned',
  payload: {
    leadId: lead.id,
    transition: 'level_changed',  // discriminator
    from: 'l1',
    to: 'l2',
    assignedRepId: lead.assignedRepId,
  },
  module: 'crm',
  tenantId: lead.tenantId,
})
// Valid transitions: created|assigned|level_changed|won|lost|stalled|contacted|merged|reactivated|cancelled
```

## Convention Checklist

| # | Convention | Check |
|---|-----------|-------|
| 1 | Use `emitOutboxEvent(tx, event)` | NEVER `publishEvent()` (deprecated) |
| 2 | Emit inside transaction | Atomic with business data |
| 3 | Event name = `{module}.{entity}.{action}` | Module prefix matches table prefix |
| 4 | Module prefix correct | `crm.`, `fin.`, `lms.`, `inf.`, `com.`, `sup.`, `cms.` |
| 5 | Payload includes entity ID | Consumer needs to look up entity |
| 6 | Transition events use discriminator | Single event type with `transition` field |
| 7 | Privacy events are separate | `crm.consent.withdrawn` is NOT a lead transition |
